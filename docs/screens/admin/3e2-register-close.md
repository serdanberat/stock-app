# 3.E.2 — Cash Register Close + Z Report

> **Status:** Locked (Phase 3.E)
> **Routes:**
> - `/cash-register/close` — Close form
> - `/cash-register/sessions/{id}/z-report` — View/reprint
> - `/cash-register/sessions` — History list

## Purpose

End-of-shift / day session close. Counts physical cash, computes variance, generates immutable Z report PDF.

## Aggregate ownership (explicit)

- **Writes** CashRegisterSession aggregate (status transition)
- Indirectly via outbox event consumer:
  - `cash_movement` CLOSING_DEPOSIT (if cash removed from drawer)
  - `cash_movement` CORRECTION (if variance, with reason captured)
  - Z report document generated by document worker (Phase 6.F)

## Z report immutability

Z report is a snapshot bound to the closed session. Once generated:
- PDF stored via DocumentStorage
- Source data (movement totals, tender breakdown, variances) captured in `z_reports` table
- **Reprint renders the SAME stored payload, NOT regenerates**
- Same payload always produces identical PDF (deterministic)

Rationale: tax/legal audit. Z reports are accounting artifacts; immutability is non-negotiable.

## Reads

- `GET /cash-register/sessions/{id}/summary`
  - Returns expected closing state:
    - `opening_float`
    - `cash_in_total` (SALE_CASH_IN + customer payments + etc.)
    - `cash_out_total` (SALE_CHANGE_OUT + refunds + payments out)
    - `expected_cash = opening + cash_in - cash_out`
    - `tender_breakdown` (sales by CASH/CARD/CUSTOMER_ACCOUNT)
    - `sale_count`, `refund_count`

## Writes

- `POST /cash-register/sessions/{id}/close`
  - Body:
    ```
    {
      counted_cash_amount,
      cash_removed_amount?,        // closing deposit to bank/safe
      variance_reason?,            // required if variance > tolerance
      variance_note?
    }
    ```
  - Idempotency-Key required
  - Atomically:
    - status = CLOSED
    - If variance: `cash_movement` CORRECTION
    - If cash_removed: `cash_movement` CLOSING_DEPOSIT
    - Z report row created (status PENDING_GENERATION)
    - Outbox event ZReportRequested triggers document worker

## Variance handling

```
variance = counted_cash_amount - expected_cash
```

`|variance| <= tenant.cash_variance_tolerance` (default ₺5):
- No reason required; variance recorded but treated as routine

`|variance| > tolerance`:
- Reason mandatory from closed set:
  - `SHORT_CASHIER_ERROR` — Kasiyer hatası (eksik)
  - `SHORT_UNKNOWN` — Eksik, sebep belirsiz
  - `OVER_CASHIER_ERROR` — Kasiyer hatası (fazla)
  - `OVER_UNKNOWN` — Fazla, sebep belirsiz
  - `MISCOUNT` — Yanlış sayım (recount yapıldı)
  - `SUSPECTED_THEFT` — Hırsızlık şüphesi (manager investigates)
- Note mandatory if SUSPECTED_THEFT

Large variance threshold (tenant config, default ₺100):
- Requires manager PIN override

## Edge cases

| # | Scenario | Behavior |
|---|---|---|
| 1 | Mid-close network failure | Idempotency-Key prevents partial close |
| 2 | Session has uncompleted Sale in AWAITING_PAYMENT | Block close: "1 satış ödeme bekliyor. Önce tamamla veya iptal et" |
| 3 | Z report generation fails | Session still CLOSED (atomic core); document worker retries; manager dashboard alert "Z report failed; manual reprint needed" |
| 4 | Reprint Z report after correction adjustment posted | Z report snapshot frozen; correction shows in NEXT session's opening calculations; reprint shows original variance |

## Audit events

- `cash_session_closed`
- `cash_session_closed_with_variance` (with amount + reason)
- `cash_session_closed_large_variance_override` (manager intervention)
- `z_report_generated`
- `z_report_reprinted` (with actor + timestamp)

## Implementation notes

- Summary load uses materialized aggregate query on cash_movements
- Z report PDF deterministic: same payload → same PDF bytes
- Reprint renders stored snapshot; does NOT recompute
- `z_reports` table schema:
  - id, tenant_id, session_id, snapshot_payload (JSONB immutable), pdf_storage_key, generated_at
