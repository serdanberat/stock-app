# Part 1 — Foundation: Identity + Catalog + Inventory

> **Status:** Locked (Phase 2E)
> **Tables:** 27
> **Migration files:** `000_extensions.sql`, `001_foundation.sql`, `002_system_seed_lookups.sql`, `003_identity.sql`, `005_catalog.sql`, `006_inventory.sql`

This is the foundation of the schema: tenants, users, stores, the product catalog, and the inventory ledger that underpins every other context.

For SQL, see the migration files. This document explains the design, decisions and invariants behind each table.

---

## 1. Identity Context (5 tables)

| Table | Purpose | Key invariants |
|---|---|---|
| `tenants` | Top of multi-tenancy hierarchy | RLS does NOT apply (drives RLS); status lifecycle TRIAL→ACTIVE→SUSPENDED→CHURNED→ARCHIVED; `anonymization_method` distinguishes irreversible vs reversible (ADR 009) |
| `users` | Operators (cashiers, managers, accountants) | `email CITEXT`; unique per tenant (excluding anonymized); `status IN (INVITED,ACTIVE,SUSPENDED,DEACTIVATED)`; user hard-delete forbidden |
| `roles` | Permission templates | `tenant_id NULL` for system roles (SUPER_ADMIN, STORE_MANAGER, CASHIER, STOCK_CLERK, ACCOUNTANT, AUDITOR); system roles protected by trigger from modification |
| `user_role_assignments` | User ↔ Role + optional store scope | `store_scope_ids UUID[]` — empty/NULL means all stores; partial UNIQUE on `(user_id, role_id)` where `revoked_at IS NULL` |
| `stores` | Physical and virtual stores | `store_type IN (PHYSICAL, VIRTUAL_IN_TRANSIT, WAREHOUSE, ONLINE)`; exactly one VIRTUAL_IN_TRANSIT per tenant; virtual cannot be deactivated |

### Highlights

- **`users.email` is CITEXT.** Case-insensitive uniqueness without `lower(email)` index hacks.
- **`user_role_assignments` is revocable, not deletable.** Past assignments retained as audit trail.
- **Stores carry `timezone`** because tenants may operate across regions; default `'Europe/Istanbul'`.
- **`feature_flags` on tenants is JSONB**, default all-conservative (`allow_admin_register_reopen: false`, `allow_negative_stock: false`, etc.). ADR 009 hardening.
- **`anonymization_method`**: `IRREVERSIBLE` default; `REVERSIBLE_PSEUDONYMIZATION` opt-in via contract addendum (ADR 009).

## 2. Catalog Context (12 tables)

| Table | Purpose |
|---|---|
| `categories` | Hierarchical product categories (max 5 levels deep) |
| `brands` | Brand master with logo, status |
| `seasons` | Season periods (e.g. SS26, AW26) with optional `starts_at`/`ends_at` |
| `attribute_types` | Color, Size, Material, Model, Gender — extensible by tenant |
| `attribute_values` | Black, M, Cotton, Slim — references `attribute_types` |
| `products` | Product model (e.g. "T-Shirt Basic") |
| `product_variants` | The unit of commerce: Product + Color + Size (the "5 pcs of Black/M") |
| `product_variant_barcodes` | Multiple barcodes per variant, scoped INTERNAL/SUPPLIER/GS1_EAN |
| `product_images`, `variant_images` | S3 image references |
| `price_lists` | Price tier (default, wholesale, VIP) |
| `variant_prices` | Time-bounded price intervals (non-overlapping via EXCLUDE) |

### Catalog invariants

- **Variant (not Product) is the unit of commerce.** Stock, sales, returns are all keyed by variant.
- **Variant uniqueness**: same product cannot have two variants with the same (color, size). Partial UNIQUE on `(product_id, COALESCE(color_id, sentinel), COALESCE(size_id, sentinel))` excluding ARCHIVED.
- **Barcode uniqueness per tenant**, not global. UNIQUE on `(tenant_id, barcode)`.
- **Barcode scope discrimination**:
  - `INTERNAL` — generated by us; flexible format (`^[0-9A-Z\-]+$`)
  - `SUPPLIER` — supplier's code, with `source` column
  - `GS1_EAN` — strict 13-digit (`^[0-9]{13}$`)
- **Barcode minimum length**: 4 characters (sanity).
- **`attribute_values.status` lifecycle**: ACTIVE/INACTIVE/ARCHIVED — INACTIVE cannot be assigned to new variants but historical references remain.
- **`products.inactive_sellable` flag**: explicit "INACTIVE but still sellable" intent — Phase 2C clarification.
- **`variant_prices` non-overlapping intervals** enforced via `EXCLUDE USING gist`. New price = INSERT a new row + update previous `valid_until`.
- **`variant_prices.reason_code`**: enum of CAMPAIGN, INFLATION_UPDATE, etc. (Phase 2D). OTHER requires `reason_notes`.

### Hierarchical category constraints

- `parent_id` self-reference, `level` integer 0–4 (5 levels max).
- Cyclic-reference prevention is application-level (recursive CTE check on insert/update).
- `CHECK (id != parent_id)` rejects trivial self-cycles.

### Image primary flags

Partial UNIQUE indexes ensure at most one `is_primary = true` per product and per variant.

## 3. Inventory Context (10 tables) — The Heart

| Table | Purpose | Append-only? |
|---|---|---|
| `reason_codes` | Lookup: STOCK_ADJUSTMENT, TRANSFER_LOSS, COUNT_VARIANCE, RETURN_REASON, VOID_REASON | No |
| `stock_movements` | The ledger. Every stock change is a row. | **YES** |
| `stock_movement_sequences` | Per `(tenant, variant, store)` aggregate_sequence allocator | No |
| `stock_balances` | Projection: current quantity + WAC per `(variant, store)` | Updated by application transaction |
| `reorder_levels` | Min/max/reorder thresholds per `(variant, store)` | No |
| `transfers`, `transfer_items` | Store-to-store transfer aggregate | No |
| `count_sessions`, `count_items` | Physical count session aggregate | No (until COMPLETED, then sealed) |
| `stock_adjustments` | Manual stock corrections | Effectively append-only (one row per adjustment, refs movement) |

### `stock_movements` — The Append-Only Heart

The single source of truth for inventory. Every stock change writes one row here. Stock balances are projections derived from this table.

```
direction       IN | OUT
movement_type   PURCHASE_RECEIPT | SALE | SALES_RETURN_IN | PURCHASE_RETURN_OUT
                | TRANSFER_IN | TRANSFER_OUT | COUNT_ADJUST_IN | COUNT_ADJUST_OUT
                | POSITIVE_ADJUSTMENT | NEGATIVE_ADJUSTMENT | WRITE_OFF
                | OPENING_BALANCE_IN
reference_type  SALE | RETURN | PURCHASE_INVOICE | PURCHASE_RETURN | TRANSFER
                | COUNT_SESSION | STOCK_ADJUSTMENT | OPENING_BALANCE
reference_id    UUID of the originating business operation
```

### Why not partitioned in MVP

The previous draft proposed RANGE partitioning by `occurred_at`. Withdrawn after realising:

- PostgreSQL partitioned tables require partition key to be part of PK
- This forces `PRIMARY KEY (id, occurred_at)` — composite
- ORM frameworks (Django ORM, SQLAlchemy, TypeORM, Prisma) often dislike composite PKs
- Foreign keys from other tables to a partitioned table only work via UNIQUE constraints that include the partition key — adding friction

Decision: **single non-partitioned table for MVP**, simple `PRIMARY KEY (id)`. Partitioning revisited in v1.1+ when row count > 5M.

Same reasoning applies to `account_movements` (Part 3).

### `stock_movements` immutability

```sql
REVOKE UPDATE, DELETE ON stock_movements FROM PUBLIC;

-- Trigger raises on any UPDATE or DELETE attempt
-- Trigger also raises if INSERT tries to reverse a row that is itself a reversal
```

To "undo" a movement, INSERT a new row with opposite direction and `reverses_movement_id = <original>`. **Reversal of a reversal is forbidden**: instead, create a new corrective movement.

### `stock_movements.aggregate_sequence`

Per `(tenant_id, variant_id, store_id)` monotonic counter, allocated via UPDATE on `stock_movement_sequences` inside the same transaction. Enables replay/debug:

> "Show me movements 1–50 for variant X in store Y; verify projection balance matches."

### `stock_movements.unit_cost_try`

Always recorded in TRY for projections. For non-TRY purchases, `unit_cost_original` + `original_currency` + `fx_snapshot_id` carry the audit trail.

### `stock_balances`

Projection table updated transactionally by application code (not by trigger). Carries:

- `quantity` — current count
- `average_cost_try` — WAC
- `total_cost_try` — generated column (`quantity * average_cost_try`)
- `last_movement_id`, `last_movement_at` for audit
- `last_reconciled_at` for nightly drift detection

**Negative quantity allowed** if `feature_flags.allow_negative_stock = true`. ADR 003 acknowledges this; reconciliation sessions may briefly produce negative balances.

### `transfers` aggregate

Two-step lifecycle: DRAFT → DISPATCHED → RECEIVED. Stock moves OUT from source store to a virtual `VIRTUAL_IN_TRANSIT` store on dispatch, then OUT of virtual → IN to destination on receive.

`transfer_items.received_quantity`: NULL during transit, set on receive. `loss_reason_code` carries the Phase 2D taxonomy:

- `LOST_IN_TRANSIT`
- `DAMAGED_IN_TRANSIT`
- `RECEIVED_SHORT`
- `PROVIDER_ERROR`
- `INTERNAL_PILFERAGE` — requires manager approval + notes (CHECK constraint)

Idempotency keys for both dispatch and receive (`idempotency_key_dispatch`, `idempotency_key_receive`) protect against accidental double-dispatch.

### `count_sessions`

Aggregate for physical inventory counts. State machine: DRAFT → IN_PROGRESS → COMPLETED. Only one IN_PROGRESS per store at a time (partial UNIQUE constraint).

**Variance formula (Phase 2C refinement):**

```
expected_qty_at_count_time = snapshot_qty + SUM(IN movements between snapshot and count)
                                          - SUM(OUT movements between snapshot and count)
variance = counted_qty - expected_qty_at_count_time
```

Adjustment movements posted on COMPLETED with reason codes (`BULUNDU`, `BULUNAMADI`, etc.).

Roadmap v2: zone/bin-scoped parallel counts for large warehouses — captured in `count_sessions` table COMMENT.

### `stock_adjustments`

Single-shot adjustments outside count sessions. Each row generates exactly one stock movement; `movement_id` is now an **explicit FK** with ON DELETE RESTRICT.

Idempotency key prevents accidental double-adjustment.

## Cross-Cutting Disciplines Reflected

| Discipline | Where applied in Part 1 |
|---|---|
| Append-only + immutability triggers | `stock_movements`, `fx_rates` (Part 3 actually), `cash_movements` (Part 3) |
| RLS via `current_tenant_id()` | Every domain table (not `tenants` itself, not pure system lookups) |
| ON DELETE policy | tenant_id → RESTRICT; child of aggregate → CASCADE; master ref → RESTRICT; audit actor → SET NULL |
| Mandatory operational actors | `transfers.dispatched_by_user_id` NOT NULL + RESTRICT; `stock_adjustments.actor_user_id` NOT NULL + RESTRICT; `count_sessions.started_by_user_id` NOT NULL + RESTRICT |
| Audit-only actors | `count_sessions.completed_by_user_id` nullable + SET NULL |
| Generated columns | `stock_balances.total_cost_try` |
| Partial UNIQUE indexes | One primary barcode/image; one default price list; one VIRTUAL_IN_TRANSIT store; one IN_PROGRESS count session per store |
| EXCLUDE constraints | `variant_prices` non-overlapping intervals |
| GIN indexes | `products.metadata`, `product_variants.attributes`, `roles.permissions`, `tenants.feature_flags` |
| Trigram indexes | `products.display_name`, `parties.display_name` (Part 3) |
| Idempotency keys | `transfers.idempotency_key_dispatch/receive`, `count_sessions.idempotency_key_complete`, `stock_adjustments.idempotency_key` |
