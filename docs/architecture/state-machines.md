# State Machines

> **Status:** Locked (Phase 2C)
> **Last updated:** 2026-05-15

This document defines the lifecycle of every aggregate. Detailed state machines are given for the 5 most critical aggregates; the remaining aggregates are summarised.

---

## 1. Sale (Detailed)

```
                  ┌─ cashier clicks "New Sale"
                  ▼
                ┌───────────┐
                │   DRAFT   │ ◄──── auto-revert from AWAITING_PAYMENT (15 min idle)
                └─────┬─────┘
                      │
        ┌─────────────┼───────────────────────┐
        │             │ "Proceed to Payment"  │ Cashier cancel
        │             ▼                       ▼
        │       ┌──────────────┐         ┌──────────┐
        │       │ AWAITING_    │────────►│  VOIDED  │
        │       │ PAYMENT      │ Cancel  │ (final)  │
        │       └──────┬───────┘         └──────────┘
        │              │
        │              │ "Complete" + atomic transaction commit
        │              ▼
        │       ┌──────────────┐
        │       │  COMPLETED   │ ◄────── final state (normal flow)
        │       │  (SEALED)    │
        │       └──────────────┘
        │
        │ 4 hours idle (DRAFT abandoned)
        ▼
   ┌──────────────┐
   │  ABANDONED   │ (final, audit trail kept)
   └──────────────┘

  COMPLETED  +  operational flags (administratively_reversed_at, ...)
            ── do NOT change state; reversal is observed in addition to COMPLETED ──
```

### Transitions

| From | To | Trigger | Pre-conditions | Side effects | Permission |
|---|---|---|---|---|---|
| ∅ | DRAFT | "New Sale" | Register session OPEN | INSERT sale row | `sales.create` |
| DRAFT | DRAFT | Cart edit | — | SaleItem CRUD; audit | `sales.modify_cart` |
| DRAFT | AWAITING_PAYMENT | "Proceed" | `items.length > 0`, `total > 0`, customer (if attached) ACTIVE, discount within limit | Status update | `sales.proceed_payment` |
| DRAFT | VOIDED | Cashier cancel | own sale | Status; audit | `sales.void_draft` |
| DRAFT | ABANDONED | Bg job: idle 4h | `last_activity < now - 4h` | Status; audit | system |
| AWAITING_PAYMENT | DRAFT | "Back" or 15 min idle | — | Status; payment attempt logged to `payment_attempts` (data preserved, never silently discarded) | system / cashier |
| AWAITING_PAYMENT | VOIDED | Cancel | — | Status; payment attempts logged | `sales.void_awaiting` |
| AWAITING_PAYMENT | COMPLETED | "Complete" + atomic OK | See atomic-transaction detail | See atomic-transaction detail | `sales.complete` |
| COMPLETED | (locked) | — | Never; reversal via flags only | — | — |

### Atomic completion transaction

```
TRANSACTION BEGIN (READ COMMITTED + row locks)
├── [1] Lock Sale (SELECT FOR UPDATE)
├── [2] Re-validate register session OPEN
├── [3] Validate sum(payments) >= total OR (sum < total AND customer_id NOT NULL)
├── [4] Finalise pricing per item (PricingService snapshot)
├── [5] FX snapshot if currency != TRY
├── [6] Cost snapshot per item (InventoryService.getCurrentAvgCost)
├── [7] Stock OUT movements per item (InventoryService.recordMovement)
├── [8] Update sale.status = COMPLETED
├── [9] INSERT SalePayment rows
├── [10] CashRegisterService movements (cash/card/transfer)
├── [11] AccountMovement (DEBIT) if partially paid
├── [12] CashRegister CHANGE_GIVEN if overpaid
├── [13] INSERT SaleDocument STUB (status PENDING_GENERATION, pdf_path NULL)
├── [14] INSERT outbox_events row (SaleCompleted)
COMMIT

AFTER COMMIT (async via outbox):
- PDF generation worker
- Receipt printer worker
- e-Document worker (v1.1+)
- Customer notification
- Reporting projector
- Audit consumer
```

### Failure handling
- Stock insufficient → ROLLBACK; cashier sees "stock issue" and adjusts cart.
- BLOCKED party → ROLLBACK; cashier asked to remove customer or take full payment.
- Credit limit exceeded → ROLLBACK + warning; manager override possible.
- Concurrent modification → SerializationFailure → retry with backoff.

### Special behaviour
- `terminal_pending = true` (e.g. card terminal awaiting callback): the abandoned-cart cleanup job does **not** abandon the sale; instead it waits for the callback and flags for manual reconciliation if the callback never arrives.
- `idempotency_key` is generated by the client and must be unique per tenant.

## 2. Return (Detailed)

```
                Cashier clicks "Return"
                          │
                          ▼
                    ┌───────────┐
                    │   DRAFT   │  Mode: RECEIPTED (sale_id) | BLIND (no sale_id)
                    └─────┬─────┘
                          │
            ┌─────────────┼──────────────┬─────────────┐
            │             │              │             │
            │             │              │             │ Cancel
            ▼             ▼              │             ▼
    ┌──────────────┐  ┌──────────────────┴───┐   ┌──────────┐
    │   AWAITING_  │  │ Direct complete     │   │  VOIDED  │
    │   APPROVAL   │  │ (under thresholds)   │   │ (final)  │
    └──────┬───────┘  └──────────┬───────────┘   └──────────┘
           │ approve              │
           ▼                      ▼
                            ┌──────────────┐
                            │  COMPLETED   │
                            │   (SEALED)   │
                            └──────────────┘
```

### Threshold logic that routes to AWAITING_APPROVAL

- RECEIPTED return whose `total > tenant.return_manager_threshold`.
- RECEIPTED return after `tenant.return_grace_days`.
- Item condition = USED.
- BLIND return whose `total > tenant.blind_return_manager_threshold`.
- BLIND return when the customer's 30-day BLIND return count exceeds `blind_return_customer_frequency_limit`.

### BLIND return hard limits (tenant-level)
- `allow_blind_return`: master flag (default false).
- `blind_return_max_amount_per_day`.
- `blind_return_max_count_per_day`.
- `blind_return_manager_threshold`.
- `blind_return_excluded_categories[]` (campaign items, final-sale items).

### Atomic completion transaction
Same shape as Sale's, mirrored:
- Stock IN movements (`SALES_RETURN_IN`).
- Refund movement (CASH out / CARD reversal / customer-balance credit / debt reduction).
- Cost snapshot:
  - RECEIPTED: copy from original `sale_item.unit_cost_try`.
  - BLIND: current WAC + `blind_return_flag = true`.
- Outbox: `ReturnCompleted`, plus `BlindReturnCompleted`, `BlindReturnApproved` and `PotentialFraudDetected` when applicable.

### Exchange flow
`Return.complete()` and a subsequent `Sale.complete()` linked by `exchange_group_id`, executed as **two separate transactions**. If the new sale fails, the customer is left with a credit balance — recoverable behaviour.

## 3. StockMovement (Append-Only, no traditional state machine)

```
            recordMovement() invoked
                      │
                      ▼
            ┌─────────────────────────┐
            │ Pre-validation          │
            │  - quantity > 0          │
            │  - type/direction match │
            │  - actor authorised      │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ TRANSACTION BEGIN       │
            │  StockBalance SELECT     │
            │   FOR UPDATE            │
            │  If OUT: balance >= qty │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ INSERT stock_movements  │
            │   (immutable)            │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ UPDATE stock_balances    │
            │ (projection)             │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ COMMIT                  │
            └─────────────────────────┘

REVERSAL: a new movement with reverses_movement_id pointing at the original.
          Reversal of a reversal is FORBIDDEN.
```

### Movement type matrix

| Type | Direction | Trigger source |
|---|---|---|
| `PURCHASE_RECEIPT` | IN | PurchaseInvoice.post() |
| `SALES_RETURN_IN` | IN | Return.complete() |
| `TRANSFER_IN` | IN | Transfer.receive() |
| `COUNT_ADJUST_IN` | IN | CountSession.complete() variance > 0 |
| `POSITIVE_ADJUSTMENT` | IN | StockAdjustment.create() |
| `OPENING_BALANCE_IN` | IN | Tenant onboarding |
| `SALE` | OUT | Sale.complete() |
| `PURCHASE_RETURN_OUT` | OUT | PurchaseReturn.post() |
| `TRANSFER_OUT` | OUT | Transfer.dispatch() |
| `COUNT_ADJUST_OUT` | OUT | CountSession.complete() variance < 0 |
| `NEGATIVE_ADJUSTMENT` | OUT | StockAdjustment (damage, gift, sample) |
| `WRITE_OFF` | OUT | StockAdjustment (dead stock) |

## 4. AccountMovement (Append-Only, no traditional state machine)

```
            FinancialService.appendJournal()
            invoked from Payment.complete(), Sale.complete(),
            PurchaseInvoice.post(), Return.complete(), or admin tool
                      │
                      ▼
            ┌─────────────────────────┐
            │ Pre-validation          │
            │  - profile exists       │
            │  - status allows direction │
            │  - currency match        │
            │  - amount > 0           │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ TRANSACTION BEGIN       │
            │  AccountBalance + Profile │
            │  SELECT FOR UPDATE       │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ INSERT account_movements │
            │   (immutable)            │
            │ UPDATE account_balances  │
            │ UPDATE account_profiles  │
            │   .credit_used           │
            └─────────┬───────────────┘
                      │
                      ▼
            ┌─────────────────────────┐
            │ COMMIT                  │
            └─────────────────────────┘
```

### Direction / type matrix

| Type | Direction | Triggered by |
|---|---|---|
| `SALE_DEBIT` | DEBIT | Sale.complete() partial pay |
| `SALE_REFUND` | CREDIT | Return.complete() to customer balance |
| `PAYMENT_RECEIVED` | CREDIT | Payment.complete() RECEIVED |
| `PAYMENT_MADE` | DEBIT | Payment.complete() MADE |
| `PURCHASE_DEBIT` | CREDIT | PurchaseInvoice.post() |
| `PURCHASE_REFUND` | DEBIT | PurchaseReturn.post() |
| `MANUAL_DEBIT` | DEBIT | Admin |
| `MANUAL_CREDIT` | CREDIT | Admin |
| `OPENING_BALANCE_DEBIT` | DEBIT | Tenant onboarding |
| `OPENING_BALANCE_CREDIT` | CREDIT | Tenant onboarding |

> **Naming note.** "DEBIT" and "CREDIT" are used from the **business**'s perspective on the party: DEBIT means "party owes us"; CREDIT means "we owe party". This is business-oriented terminology, not formal double-entry accounting terminology where the meaning is from the accountant's side. Helper functions `party_owes_us(...)` and `we_owe_party(...)` are provided to avoid confusion at the call site.

## 5. RegisterSession (Detailed)

```
              Cashier "Open Register" + opening_float
                            │
                            ▼
                      ┌──────────┐
                      │   OPEN   │ ◄────────┐
                      │          │          │
                      │  Sales,  │          │
                      │  Returns,│          │
                      │  Cash    │          │
                      │  in/out  │          │
                      └────┬─────┘          │
                           │                │
                           │ "Close Day"    │ Manager
                           ▼                │ override
                      ┌──────────┐          │ (rare,
                      │ CLOSING  │          │  feature-flag
                      │          │          │  gated)
                      │ Grace    │          │
                      │ period:  │          │
                      │ 10 min   │          │
                      │          │          │
                      │ ✗ New    │          │
                      │   sales  │          │
                      │ ✓ Finish │          │
                      │   existing │         │
                      │ ✓ Counting │         │
                      └────┬─────┘          │
                           │                │
                           │ Count OK +     │
                           │ Z report stub  │
                           │ + atomic       │
                           ▼                │
                      ┌──────────┐          │
                      │  CLOSED  │──────────┘ (super_admin only,
                      │  SEALED  │              with co-signature)
                      └──────────┘
```

### CLOSING grace period
- Set when "Close Day" clicked.
- During CLOSING:
  - ✗ No new Sale or Return may start.
  - ✓ Existing DRAFT / AWAITING_PAYMENT sales may complete (within 10 minutes).
  - ✓ Cash counting UI active.
- After 10 minutes:
  - Open sales whose `terminal_pending = false` → ABANDONED.
  - Open sales whose `terminal_pending = true` → flagged for manual reconciliation; **not** abandoned.
  - System then attempts finalisation.

### Atomic close transaction
- Re-validates state, computes expected cash, records variance movement if any.
- Allocates Z report number only on commit (no sequence gaps).
- Generates ZReport stub (status PENDING_GENERATION, pdf_path NULL).
- Outbox: `RegisterClosed`, `ZReportGenerated`, optionally `CashDiscrepancyDetected`.
- Async after commit: PDF generation, owner notification, reporting projection.

### Z report number sequence
- Per `(tenant_id, store_id, year)`.
- Allocated by an UPDATE inside the same transaction. Rollback undoes the increment — no gaps.
- Audit trail: `z_report_sequence_audit`. Nightly integrity check verifies committed Z reports cover every allocated number.

### CLOSED → OPEN (administrative re-open)
Tightly restricted:
- Feature flag `tenant.allow_admin_register_reopen` (default off).
- Super admin only.
- No subsequent session may exist for the same register.
- No invoices may have been issued after the Z report.
- Reason text ≥ 100 chars.
- Co-signature from a second super admin.
- Anthropic support ticket required.
- The original Z report is marked INVALIDATED (but retained); a new Z report will be generated on the next close.

## Summary State Machines (Remaining Aggregates)

### Identity context

**Tenant**
```
TRIAL → ACTIVE → SUSPENDED → CHURNED → ARCHIVED (anonymised cold storage)
                                            ↓
                                  (physical purge only after 10 years)
```
- Hard deletion forbidden (see ADR 007).
- ARCHIVED applies PII anonymisation but keeps commercial records.

**User**
```
INVITED → ACTIVE ↔ SUSPENDED → DEACTIVATED
```
- DEACTIVATED is soft; historical references preserved.
- The last super-admin user cannot be DEACTIVATED.

**Role**
```
ACTIVE ↔ DISABLED → ARCHIVED
```
- System roles (6) cannot be archived or disabled.

**Store**
```
ACTIVE ↔ INACTIVE → ARCHIVED
(special: VIRTUAL_IN_TRANSIT — uneditable)
```
- ARCHIVED only when stock = 0 and no pending transfers.

### Catalog context

**Product**
```
DRAFT → ACTIVE ↔ INACTIVE → DISCONTINUED → ARCHIVED (auto after 6 months of zero stock)
```
- `inactive_sellable` flag controls whether INACTIVE allows barcode sale.
- DISCONTINUED forbids new purchase invoices; existing stock remains sellable.

**Category**
```
ACTIVE ↔ DISABLED → ARCHIVED
```
- Hierarchy max 5 levels; cyclic refs forbidden.

**PriceList**
```
DRAFT → ACTIVE → EXPIRED
       └→ SUSPENDED (temporary hold)
```
- MVP has a single `DEFAULT_RETAIL`, ACTIVE indefinitely.
- Variant prices are append-only inside the list (new row + close previous `valid_until`).

### Inventory context (Transfer, CountSession, StockAdjustment)

**Transfer**
```
DRAFT → DISPATCHED → RECEIVED
       └→ CANCELLED
```
- DISPATCHED uses virtual `IN_TRANSIT` store; cannot be cancelled, only reversed.
- Loss reason codes: `LOST_IN_TRANSIT`, `DAMAGED_IN_TRANSIT`, `RECEIVED_SHORT`, `PROVIDER_ERROR`, `INTERNAL_PILFERAGE`.

**CountSession**
```
DRAFT → IN_PROGRESS → COMPLETED
                    └→ CANCELLED
```
- Only one IN_PROGRESS per store (MVP).
- Roadmap v2: zone/bin-scoped parallel counts.

**StockAdjustment**
```
(single-shot) → POSTED
```
- No DRAFT state; created and committed atomically.

### Purchasing context

**PurchaseInvoice**
```
DRAFT → POSTED
      └→ CANCELLED
POSTED + operational flags (administratively_reversed_at, ...) for rare admin reversal
```

**PurchaseReturn**: same shape as PurchaseInvoice.

### Party / Financial

**Party**
```
ACTIVE ↔ INACTIVE → BLOCKED ↔ ACTIVE (with manager approval)
```

**AccountProfile**
```
NORMAL ↔ WATCH → BLOCKED ↔ NORMAL (with manager approval)
                      └→ CLOSED (terminal)
```
- BLOCKED forbids DEBITs but allows CREDITs (payments).
- CLOSED forbids all new entries.

**Payment**
```
DRAFT → COMPLETED → REVERSED  (or)
                  → PARTIALLY_REVERSED (cumulative) → REVERSED (when sum reaches original)
     ├→ FAILED (terminal — retry = new payment with new idempotency key)
     └→ CANCELLED
```
- Reversal of reversal forbidden.

### CashRegister

**CashRegister** (the physical till)
```
ACTIVE ↔ INACTIVE → ARCHIVED
```

### FX

**Currency**: `ACTIVE ↔ INACTIVE` (never deleted; TRY always ACTIVE).
**FxRateSource**: `ACTIVE ↔ INACTIVE` (MANUAL always active).
**FxRate**: append-only (no states).
**FxSnapshot**: append-only (no states).
