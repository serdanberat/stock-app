# ADR 005 — Asynchronous Document Generation

> **Status:** Accepted
> **Date:** 2026-05-15

## Context

A retail POS sale, return or purchase invoice produces multiple downstream artefacts:

- A printable receipt or invoice (PDF).
- A file upload to object storage.
- An e-Arşiv / e-Fatura submission to a regulatory provider (Turkey, v1.1+).
- A print job dispatched to the store's receipt printer.
- Optionally, a customer-facing email or SMS notification.

If these activities run **inside** the atomic domain transaction:
- An S3 timeout rolls back a successful sale.
- A PDF rendering crash rolls back a successful sale.
- e-Fatura provider latency makes the POS feel slow.
- A receipt printer being offline blocks the customer at the counter.

These outcomes are unacceptable. A completed sale must remain completed even when downstream artefacts are delayed.

## Decision

Document generation is **asynchronous after commit**:

1. The atomic domain transaction (Sale.complete(), Return.complete(), PurchaseInvoice.post(), RegisterSession.close()) writes a **document stub** with `status = PENDING_GENERATION` and `pdf_path = NULL`.
2. The transaction commits. The user-visible result completes.
3. The outbox event (`SaleCompleted`, `ReturnCompleted`, `PurchaseInvoicePosted`, `RegisterClosed`) is published.
4. Async workers consume the event and produce the artefacts:
   - **PDF generator worker** renders the PDF, uploads to object storage, updates the stub: `status = READY`, `pdf_path = ...`.
   - **Receipt printer worker** dispatches the print job to the store's printer; retries on failure.
   - **e-Document worker** (v1.1+) submits to the e-Arşiv / e-Fatura provider; updates with returned UUID.
   - **Notification worker** sends customer email/SMS if configured.

### Document stub schema

```
sale_documents / return_documents / purchase_invoice_documents (
  id, sale_id (or return_id, etc.),
  document_type           ENUM('RECEIPT','INVOICE','E_ARSIV','E_FATURA','Z_REPORT'),
  document_number         VARCHAR NULL,   -- assigned at commit when sequence-based
  pdf_path                VARCHAR NULL,   -- S3 path, filled after generation
  status                  ENUM('PENDING_GENERATION','GENERATING','READY','FAILED'),
  e_document_uuid         VARCHAR NULL,
  e_document_status       ENUM(...) NULL,
  printed_at              TIMESTAMPTZ NULL,
  generated_at            TIMESTAMPTZ NULL,
  attempt_count           INT DEFAULT 0,
  last_error              TEXT NULL
);
```

### File uploads attached **to** documents

For inbound documents (a scanned supplier invoice attached to a `PurchaseInvoice`), the **file is pre-uploaded** before the transaction:

1. User picks the file in the UI; client uploads to a `pending_document_uploads` table / S3 path.
2. The atomic transaction references the pre-uploaded file.
3. Commit succeeds without performing the upload.

This keeps the transaction fast and predictable, while ensuring the file exists by the time the document is referenced.

### Retry semantics

| Worker | Retry policy | After max attempts |
|---|---|---|
| PDF generator | Exponential backoff, 3 attempts | Mark `FAILED`, alert admin, user can re-trigger |
| Printer | Retry queue with backoff | Allow manual reprint from UI |
| e-Document | Provider-specific policy | Queue for manual reconciliation |
| Notification | 5 attempts | Drop silently after; in-app notification only |

### Z report PDF generation

The Z report number is allocated **at commit** (so there are no sequence gaps), and the PDF stub is written at commit. The actual PDF is generated by the same async worker pattern.

## Rationale

- **The sale is the truth, the document is a side effect.** Conflating the two punishes customers for unrelated technical failures.
- **POS responsiveness.** Async generation keeps end-of-sale UX under 200ms even with large receipt content.
- **External providers fail.** e-Fatura providers have outages. The user must not feel them.
- **Storage providers fail.** S3 has occasional timeouts. The sale must not roll back because of one.
- **Recoverability.** If a worker fails, the stub captures the failure; users and admins can act.

## Consequences

**Positive:**
- Fast, reliable POS UX.
- Failures are visible, addressable, retryable.
- The user can always re-print or re-generate a document on demand.
- Compliance integrations (e-Fatura) can fail without endangering the sale.

**Negative:**
- The end-to-end "sale → receipt in hand" path is now multi-step; we must show appropriate UI state ("Receipt printing…").
- Slightly more storage and tracking overhead. Negligible.

## Alternatives Considered

- **Synchronous generation inside transaction** — rejected. Every reason above.
- **Sync generation outside transaction, before user feedback** — rejected. Still couples user feedback to flaky external resources.
- **Skip document stub, generate fully async without persistence** — rejected. We need a row to track failures and retries.
